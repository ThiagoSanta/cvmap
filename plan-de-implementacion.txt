FASE 1 — Capa de infraestructura externa
------------------------------------------
Objetivo: poder ir de "nombre de ciudad" a "lista cruda de empresas".

- src/config/database.js
  Conexión a SQLite con better-sqlite3. Ejecuta schema.sql si las tablas 
  no existen.

- src/config/env.js
  Carga y valida variables de entorno desde .env.

- src/config/overpass-client.js
  Construye la query Overpass (bbox). Maneja fallback de servidor 
  (OVERPASS_API_URL -> OVERPASS_API_URL_FALLBACK).

- src/config/nominatim-client.js (NUEVO)
  Geocodifica ciudad -> lat/lon. Agregar a .env: 
  NOMINATIM_API_URL=https://nominatim.openstreetmap.org/search

- Pendiente para decisions.md (agregar cuando se retome):
  "Geocoding de ciudad vía Nominatim, resultado cacheado. Evaluar tabla 
  geocache chica (ciudad -> lat/lon) para evitar re-geocodificar la misma 
  ciudad en cada búsqueda."


FASE 2 — Módulo companies (dominio + aplicación + infraestructura)
------------------------------------------
- domain/company.entity.js
  Validación básica de forma del objeto empresa.

- domain/company.repository.js
  Interfaz: findByOsmId, save, findByBoundingBox, etc.

- infrastructure/sqlite-company.repository.js
  Implementación real con better-sqlite3.

- application/search-companies.usecase.js
  Orquesta: geocodificar ciudad -> definir bbox según radio -> chequear

FASE 3 — Módulo tracking
------------------------------------------
Mismo patrón de capas que companies.

- application/update-status.usecase.js
  Cambia status en tabla tracking. Crea el registro si no existe para 
  esa empresa.

- application/add-event.usecase.js
  Inserta en tracking_events (envío de CV, respuesta, entrevista, nota).

- Rutas:
  PATCH /api/tracking/:companyId/status
  POST /api/tracking/:companyId/events
  GET /api/tracking/:companyId


FASE 4 — Servidor Express
------------------------------------------
- server.js
  Arma la app, monta rutas de ambos módulos, sirve public/ como estático.

- src/shared/middlewares/error-handler.js
  Captura errores de Overpass/Nominatim caídos y devuelve mensaje claro 
  al frontend (no un 500 pelado).

- src/shared/errors/
  Clases de error propias (ej: GeocodingError, OverpassError).

NOTA — Timeout de cara al usuario en GET /api/companies (RESOLVER EN FASE 4):
  Con 3 servidores × 60s de timeout cada uno, el peor caso es ~3 minutos
  antes de lanzar OverpassError. Esto es inaceptable para un browser (muchos
  navegadores y proxies HTTP cortan antes de los 3 minutos).
  Opciones a evaluar al implementar el endpoint:
  a) Timeout de cliente reducido para el contexto HTTP (p.ej. 20s/servidor)
     con el timeout largo reservado para scripts de ingesta.
  b) Respuesta inmediata con datos de caché + polling para resultado fresco
     (si hay caché disponible, devolver primero y refrescar en background).
  c) Mensaje de progreso via Server-Sent Events o WebSocket mientras se
     recorre la cadena de fallbacks.
  d) Devolver error HTTP 503 rápido con Retry-After si ningún servidor
     responde en N segundos, dejando que el frontend muestre estado de carga.

ORDEN DE DEPENDENCIAS
------------------------------------------
No se puede cachear (Fase 2) sin el cliente de Overpass/Nominatim 
(Fase 1). No se puede hacer tracking (Fase 3) sin empresas guardadas. 
El frontend (Fases 5-8) necesita que el backend ya responda.
